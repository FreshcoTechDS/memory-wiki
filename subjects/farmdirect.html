# FarmDirect — Backend Logic & Service Architecture

## DoorDash Pattern → FarmDirect Translation

| DoorDash | FarmDirect | Key Difference |
|----------|-----------|----------------|
| Restaurant lists menu | Farmer lists **harvest inventory** | Finite, seasonal, spoils — not made-to-order |
| Customer orders meal | Customer orders **produce by weight/unit** | Bulk + consumer portions coexist |
| Restaurant accepts & cooks | Farmer accepts & **harvests/packs** | Time-to-ready varies (hours, not minutes) |
| Dasher picks up at restaurant | Driver picks up at **farm** | Rural locations, longer distances |
| Real-time tracking | Real-time tracking | GPS critical for rural routes |
| Dasher drops off | Driver delivers to consumer | May batch multiple farm pickups |

# FarmDirect — Farmer-to-Consumer Marketplace

## Overview

A marketplace connecting local farmers directly with consumers, handling listings, orders, payments, and eventually delivery. Farmers list their harvest; consumers browse and buy. The platform handles payments via Stripe Connect (escrow), real-time chat via Firestore, and delivery orchestration as a Phase 2 plugin.

## Architecture (Google Cloud — us-central1)

### Serverless-First on Cloud Run

Every backend service runs as its own Cloud Run container:

| Service | Responsibility |
|---------|---------------|
| **Auth Service** | Identity Platform integration, JWT issuance, OAuth 2.0 |
| **Listing Service** | Product CRUD, search, filtering, inventory management |
| **Order Service** | Cart → confirmation state machine, order lifecycle |
| **Payment Service** | Stripe Connect integration, escrow, farmer payouts |
| **Notification Service** | Push (FCM), email (SendGrid), SMS alerts |
| **Delivery Service** | *(Phase 2)* Driver matching, route optimization — new Pub/Sub subscriber, zero code changes to existing services |

### Event-Driven with Pub/Sub

Services publish events, not direct calls:
